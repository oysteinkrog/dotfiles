# Implementation Ops

Estate plans fail in implementation as often as they fail in legal design.
This file turns the plan from "documents we intend to sign" into "a system that will
actually work in the real world."

---

## Core Rule

The planning layer and the implementation layer are separate.

- A drafted revocable trust is not a funded trust.
- A good dispositive scheme is not an updated beneficiary form.
- A named executor is not a reachable executor with the right documents.
- A signed will is not a completed plan if deeds, account titles, and institutions still
  point somewhere else.

---

## Required Implementation Tracks

### 1. Signing Readiness

Use `deliverables/signing-readiness-checklist.md` to capture:

- which documents must be signed
- where and when the ceremony happens
- witness / notary / self-proving requirements
- capacity / undue-influence safeguards
- which items must be physically present at signing
- what must be done immediately after signing

### 2. Institution and Beneficiary Updates

Use:

- `deliverables/institution-contact-matrix.md`
- `deliverables/beneficiary-change-packet.md`

Capture:

- institution name
- contact path / portal / advisor
- account or policy involved
- current control setting
- intended control setting
- whether legal documents are required for the change
- whether medallion signature / notary / wet-ink / court paperwork is likely

### 3. Titling and Funding Proof

Use:

- `deliverables/implementation-ledger.md`
- `deliverables/funding-proof-log.md`

The ledger is the queue.
The proof log is the evidence that the queue was actually completed.

Examples:

- deed into revocable trust recorded
- brokerage TOD updated
- IRA contingent beneficiaries updated
- ILIT ownership and beneficiary assignments completed
- LLC membership interests assigned per plan

### 4. Business Continuity

If the user owns or runs an operating business, also produce:

- `deliverables/business-continuity-activation.md`

The question is not only "who inherits the business?"
It is "who can run payroll, approve wires, communicate with staff, and preserve value the
day after death or incapacity?"

## Operational Lanes

### Lane A: Pre-signing

- confirm facts
- confirm current law
- finalize attorney questions
- identify missing documents
- resolve institution-specific blockers

### Lane B: Signing ceremony

- witness / notary logistics
- simultaneous-presence issues
- self-proving affidavit handling
- capacity and undue-influence safeguards

### Lane C: Immediate post-signing

- store originals
- distribute copies where appropriate
- update letter of instruction
- start beneficiary / title / funding changes

### Lane D: Proof and maintenance

- capture confirmations
- record dates completed
- note institutions that rejected or delayed changes
- set maintenance reminders

---

## Institution Friction Checklist

Expect operational friction for:

- 401(k) and employer plan vendors
- old life insurance carriers
- transfer agents for private stock
- private funds requiring internal approval
- county recorders
- banks with POA acceptance issues
- family LLCs with neglected books

Do not assume the institution will implement the legal intent cleanly just because the
documents are valid.

---

## Required Human Escalations

Escalate to attorney / CPA / advisor when the implementation step requires:

- custom assignment language
- entity consents
- deed preparation / recording strategy
- retirement trust beneficiary drafting
- tax elections
- private-fund or operating-agreement interpretation

---

## Anti-Pattern

Do not leave the user with a "great plan" and no concrete path to:

- sign it safely
- fund it
- prove it was funded
- update institutions
- keep the business alive

# Quality Bar

## Small-scope update

Minimum acceptable:

- explicit scope window
- correct version/date references
- at least one representative commit link
- no fake release links

## Standard repo rebuild

Minimum acceptable:

- full version timeline for the requested window
- capability-wave grouping
- representative commit links
- tracker/workstream integration if available
- clear notes for agents

## Large-history reconstruction

Minimum acceptable:

- durable research memo
- explicit chunking strategy
- coverage ledger with no silent gaps
- changelog updated after every chunk
- release-vs-tag correctness
- direct commit URLs
- sections that help another agent orient without diff-mining

## Ultra-large history reconstruction

Minimum acceptable:

- `CHANGELOG_RESEARCH/` multi-pass workspace
- explicit coverage ledger
- chunk files with distilled findings, not raw logs
- generated version spine and tracker extraction artifacts
- clustering or similar wave-discovery aid
- final changelog updated during the process, not only at the end

## Signs the changelog is actually strong

- another agent can identify the 3-5 biggest evolution waves quickly
- the version spine is easy to trust
- links are useful, not decorative
- workstream links explain intent, not just existence
- the document saves time on future archaeology

## Signs it is still weak

- vague summaries like "many improvements"
- too many raw commits and too little synthesis
- sections that could apply to any repo
- dates that rely on relative phrasing
- missing explanation of why a wave mattered
